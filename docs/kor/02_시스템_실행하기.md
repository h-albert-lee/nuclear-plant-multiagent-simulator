# 시스템 실행하기

## 실행 모드

| 모드 | 명령 | 용도 |
|---|---|---|
| **로컬 + mock** | `make smoke` 또는 `python -m src.main --config configs/config.yaml --mock-llm` | LLM 비용 0, 빠른 검증 |
| **로컬 + 실제 LLM** | `python -m src.main --config configs/config.yaml` + 환경변수로 `LLM_PROXY_URL` 지정 | 개발 디버깅 |
| **Docker** | `make build && make up` | 정식 배포, 격리 보장 |

## 로컬 실행 (Docker 없이)

```bash
make install                      # 가상환경 .venv 생성 + 의존성 설치
.venv/bin/python -m src.main \
    --config configs/config.yaml \
    --scenario scenarios/normal_baseline.yaml \
    --max-ticks 200 \
    --mock-llm \
    --log-level INFO
```

옵션 설명:
- `--scenario`: YAML 시나리오 파일. 기본값은 `configs/config.yaml.run.scenario`.
- `--max-ticks`: 최대 tick 수. 안전기능 손상이나 외부 stop 신호가 더 먼저 오면 그쪽에서 종료.
- `--mock-llm`: 모든 에이전트의 LLM provider 를 `mock` 으로 강제 (정형 JSON 응답 캐닝).
- `--log-level`: `DEBUG | INFO | WARNING | ERROR`.

## Docker 실행

### 첫 빌드

```bash
cp .env.example .env
# 편집: 사용할 LLM provider 의 API 키만 채워도 됨
```

`docker compose` 사용:
```bash
docker compose build
docker compose up -d
docker compose logs -f simulator
```

또는 `make` shortcut:
```bash
make build
make up
make logs
```

### 정지

```bash
make down                         # docker compose down — 컨테이너 제거, runs/ 는 호스트에 남음
```

### 헬스체크

```bash
curl http://localhost:8080/run/info
# {"run_id":"...","current_tick":42,"status":"running"}

curl http://localhost:9000/healthz   # llm_proxy 직접 노출 안 되어 있음 (internal network)
                                     # 컨테이너 내부에서만 가능:
docker compose exec llm_proxy curl http://localhost:9000/healthz
```

## 시나리오 선택

기본 네 개:

- `scenarios/normal_baseline.yaml` — 정상 운전. 어택 없으면 안전기능 6/6 intact 유지.
- `scenarios/sbo_v1.yaml` — Station Black-Out. tick 50 에 외부 전원 상실 → AOP/EOP 진입.
- `scenarios/loca_small.yaml` — 소파단 LOCA. tick 30 에 RCS 누설 시작 → 자동 SIAS/CIAS → tick 90 SCRAM → tick 200 CSAS/고방사선. RCS 무결성 lost 로 자동 종료.
- `scenarios/sgtr_v1.yaml` — SGTR. tick 40 에 SG 전열관 파단 → 2차 측 방사선 알람 → tick 150 SCRAM + MSIS.

직접 작성:
```yaml
# scenarios/my_scenario.yaml
scenario_id: my_scenario
duration_ticks: 1500
initial_state:
  vars:
    reactor_thermal_power: 100.0
    pressurizer_pressure: 15.5
    # ... (configs/config.yaml 의 초기값 참고)
  systems:
    RPS: { status: running }
  procedures:
    active: NOP-12
events:
  - tick: 100
    type: "your event"
    effect:
      vars: { pressurizer_pressure: 16.5 }
      alarms:
        ALM-PRZ-PRESS-HIGH:
          state: active
          severity: warning
          suppress_policy: conditional
          setpoint_tier: high
action_effects: []
```

`event.effect` 가 받는 키:
- `vars: {var_id: float}` — 변수 직접 설정
- `systems: {sys_id: {status, safety_logic_active}}` — 시스템 상태 변경
- `alarms: {alarm_id: {state, severity, was_critical, suppress_policy, setpoint_tier, description}}` — 알람 생성/수정
- `procedures: {active, step}` — 절차서 전환

## 동시에 여러 run 돌리기

각 run 은 별도 `run_id` (UUID) 디렉토리에 출력하므로 동일 시스템에서 여러 시뮬을 *순차* 실행할 수 있습니다. 단 Docker compose 는 기본적으로 컨테이너 이름이 고정되어 있어 *동시* 실행은 불가 — port 8080 도 단일 점유. 동시 실험이 필요하면 docker-compose project name 을 다르게 띄우거나 호스트 port mapping 을 분리하세요:

```bash
COMPOSE_PROJECT_NAME=run1 docker compose -f docker-compose.yaml up -d
# 다른 터미널:
COMPOSE_PROJECT_NAME=run2 SIM_PORT=8081 docker compose ... # 별도 compose 작성 필요
```

## Tick speed 조정

`configs/config.yaml`:
```yaml
run:
  tick_seconds: 30        # sim 1 tick 이 표현하는 모의 시간 (30 sec). 변경 시 알람 임계 영향.
  sim_time_scale: 0.1     # wall-time 압축. 0.1 = sim 30s 가 wall-time 3s.
```

`sim_time_scale` 을 `0.01` 로 하면 매우 빠르게 돌고, `1.0` 으로 하면 실시간으로 진행됩니다. 모의 실험은 `0.05~0.1` 이 무난.

## 종료 조건

자동 종료 트리거 (먼저 만나는 것):
1. `tick >= max_ticks` (기본 2000)
2. 6 CSF 중 어느 하나라도 `lost` 상태로 전환
3. 외부에서 `POST /run/stop` 호출

`run_summary.json.termination_reason` 에 사유 기록됨.
