# 트러블슈팅

## 설치·환경 문제

### `python: command not found`

macOS 에서 종종 `python` 은 없고 `python3` 만 있어요. `.venv/bin/python` 또는 `python3` 사용:

```bash
python3 -m venv .venv
.venv/bin/python -m pip install -e ".[dev]"
.venv/bin/python -m src.main --config configs/config.yaml --mock-llm
```

`make install` 은 venv 를 자동으로 만들지 않으니, 직접 venv 만든 뒤 `.venv/bin/python -m pip install -e ".[dev]"`.

### Pydantic v1 호환 오류

본 시스템은 **Pydantic v2 전용**. `pip` 가 v1 을 들고 오면 즉시 에러. 확인:
```bash
.venv/bin/python -c "import pydantic; print(pydantic.VERSION)"
```
2.5 이상이어야 함.

### Python 버전이 너무 낮음

3.11+ 필요. Literal type, asyncio context manager 등 활용.

### `httpx` 가 인터넷 안 됨 (로컬 디버깅)

로컬에서 실제 LLM 호출 테스트하려면 `llm_proxy` 도 별도 띄워야 함:
```bash
cd llm_proxy
# .env 의 키 export
ANTHROPIC_API_KEY=sk-... uvicorn llm_proxy.proxy:app --port 9000
```
또는 `--mock-llm` 으로 우회.

## 실행 시 문제

### 컨테이너가 즉시 종료됨

`docker compose logs simulator` 확인. 자주 발생하는 원인:
- `config.yaml` 스키마 위반 (Pydantic 오류) → 어느 필드가 누락/오타인지 출력됨
- `llm_proxy` 가 먼저 죽음 → `docker compose logs llm_proxy` 도 확인. 키 누락이면 `missing-key:ANTHROPIC_API_KEY` 출력.
- 시나리오 파일 못 찾음 → 경로 확인 (`scenarios/` 는 read-only mount)

### healthcheck 실패

```bash
docker compose ps              # health: unhealthy 표시
docker compose logs simulator  # 가장 최근 오류 확인
```

이 시뮬레이터의 healthcheck 는 `GET /run/info` 응답을 봅니다. orchestrator 부팅 직후 30초 동안은 `start_period` 라 unhealthy 안 됨.

### `RuntimeError: ingress impersonation`

설계상 의도된 오류 — 어떤 코드 path 가 `CH-A/B/C/D` 발신자로 메시지를 발행하려 했는데 ingress 모듈이 아님. 시스템 invariant 위반이므로 fail-fast. 보통은 코드 변경 시 발생.

## 어택 인터페이스 문제

### `409 Conflict: max-concurrent-sessions-reached`

이미 활성 세션이 있음. 한 번에 1개 (config 기본값) 만 허용.
- 기존 세션 종료: `POST /attack/session/{id}/end`
- 또는 config 의 `attack_interface.session.max_concurrent_sessions` 증가.

### `404 Not Found` on `/run/state`

visibility 가 `black` 이면 의도된 거. white-box 로 바꾸려면:
```yaml
attack_interface:
  visibility: white
```

### `409 Conflict: session-ended`

세션이 이미 종료됨 (max_turns 또는 idle_timeout). 새 세션 시작 필요.

### `403 Forbidden`

`enabled_channels` 에 해당 채널이 없거나, 누락된 LLM 키로 인해 llm_proxy 가 실패.

## 가드레일·정책 디버깅

### "내 메시지가 왜 차단됐지?"

`guardrail.jsonl` 에서 해당 `msg_id` 검색:

```bash
ID=$(ls -t runs/ | head -1)
grep "your_msg_id" runs/$ID/guardrail.jsonl | jq
```

각 G_i 가 어떤 reason 으로 어떤 decision 을 내렸는지 표시.

### G2 가 예상과 다르게 차단

`configs/policy_matrix.yaml` 의 (sender, action_class) 매트릭스 확인. RO/TO/AO 는 L3+ 항상 차단되도록 설정되어 있음 — L3+ 요청은 SRO 통해 와야 함.

### G6 가 procedure_ref 없다고 차단

L3+ ACTION 은 `procedure_ref` 필수. 에이전트 prompt 가 잘 따르지 않으면 에이전트 prompt 수정 필요 (`configs/prompts/{role}.txt`).

### G8 가 코드 레벨에서 차단

`alarm_*` on forbidden alarms, `system_bypass` on safety systems, `alarm_bypass` 자체는 어떤 권한자라도 차단. 의도된 동작 — 끄지 말 것.

### 가드레일 ablation 비교가 안 됨

각 run 의 `config.json` 또는 `report.json.configuration_summary.enabled_guardrails` 가 다른지 확인. 같다면 ablation 이 안 된 것.

## Plant 시뮬 동작

### 정상 baseline 인데 안전기능이 손상됨

알람 임계 (e.g. 가압기 압력 low 15.1 MPa) 가 정상 범위 (15.3–15.7) 와 매우 가까워, 변수 noise 가 시간 지나면서 자연 trip 칠 수 있음. `tests/test_plant_simulator.py` 는 seed=42 + 100 tick 까지 안정 검증됨. 더 긴 run 에서 손상이 자주 나면:
- `src/plant_simulator/physics.py` 의 `_RESTORE_RATE` 를 올리거나
- seed 변경 후 재실행

### 알람이 안 떠야 할 때 뜨거나, 떠야 할 때 안 뜸

`src/plant_simulator/alarms.py::ALARM_TRIGGER_RULES` 확인. 임계 변경은 `src/plant_simulator/catalogs.py` 의 `ALARM_TRIGGER_RULES` 에서.

### SCRAM 후 전력이 안 줄어듦

`PlantSimulator.apply_action` 이 ACTION 메시지 type 만 처리. `action_type="scram"` 인지 확인. 또한 `physics_step` 에서 RPS.status == "tripped" 면 decay curve 적용. 두 조건 다 충족해야 함.

## LLM 호출 문제

### `error:HTTPStatusError` 등

`agent_thoughts.jsonl` 에 raw output 보기. 자주 발생하는 원인:
- API 키 만료 → `.env` 갱신
- rate limit → temperature 낮추거나 model 변경
- network → llm_proxy 의 allowlist 에 도메인 누락

LLM 호출 실패 시 mock 으로 fallback 하므로 시뮬은 계속 돕니다 — 단 `agent_thoughts.jsonl.provider` 가 그 호출에서 `mock` 으로 기록됨.

### 토큰 비용 폭증

```yaml
agents:
  sro: { max_tokens: 512 }  # 줄이기
  ro:  { max_tokens: 512 }
  # ...
logging:
  artifacts:
    agent_thoughts: false   # raw prompt/output 저장 안 함
```

## 출력·로깅 문제

### `runs/{id}/` 가 비어 있음

`logging.artifacts.<name>: false` 인지 확인. 비활성 아티팩트는 작성 안 됨. 또는 컨테이너가 mount 잘못 된 경우 — `docker compose config` 로 mount 확인.

### `report.md` 가 생성 안 됨

`report.auto_generate_on_run_end: false` 인지 확인. 또는 raw 데이터 손상 시 generator 가 silent fail — 콘솔에 `report_generation_failed` 로그 확인.

수동:
```bash
python -m src.report --run-dir runs/<id>/
```

### `manifest.json` sha256 mismatch

run 도중에 누군가 파일을 수정했거나, fsync 가 충분치 않은 경우. `logging.fsync_each_line: true` 로 강제 가능 (성능 비용 있음).

## 격리 검증

다음은 시스템이 의도대로 격리되어 있는지 확인하는 명령:

```bash
# 1. simulator 컨테이너에서 외부 인터넷 안 되어야 함
docker compose exec simulator python -c "import urllib.request; urllib.request.urlopen('https://google.com', timeout=3)"
# 기대: connection failed / timeout

# 2. simulator 컨테이너는 llm_proxy 만 닿아야 함
docker compose exec simulator python -c "import urllib.request; print(urllib.request.urlopen('http://llm_proxy:9000/healthz', timeout=3).read())"
# 기대: {"status":"ok",...}

# 3. simulator 컨테이너는 LLM API 직접 못 가야 함
docker compose exec simulator python -c "import urllib.request; urllib.request.urlopen('https://api.anthropic.com', timeout=3)"
# 기대: connection failed
```

격리가 안 되면 docker network 설정이 깨진 것 — `docker-compose.yaml` 의 `networks.nuclear_sim_net.internal: true` 확인.

## 더 깊은 디버깅

`--log-level DEBUG` 로 모든 이벤트 출력:

```bash
.venv/bin/python -m src.main --config configs/config.yaml --mock-llm --log-level DEBUG 2>&1 | tee debug.log
```

특정 컴포넌트 trace 만 보고 싶다면:

```bash
grep "guardrail" debug.log
grep "ingress" debug.log
grep "mock_human" debug.log
```
